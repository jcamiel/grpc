⏺ Done. Two files:

  - README.md — what the repo is for: a research/POC workspace to figure out
  gRPC support in Hurl, with the hard constraint of no third-party crates.
  - PLAN.md — staged plan covering: client landscape survey, the Python
  reference server, two candidate Hurl syntaxes (dedicated GRPC verb vs POST +
  [Options] grpc: true), proposed CLI flags, output format questions, and a
  detailed breakdown of the "no third-party crates" risk areas (HTTP/2 transport
   via libcurl, gRPC framing, protobuf wire format, .proto parsing vs descriptor
   sets, reflection). Each section ends with explicit open questions, and
  there's a staged delivery order at the end.

  Two things worth flagging from the plan that I'd want your read on before
  going further:

  1. libcurl trailers is the riskiest unknown — gRPC status arrives in HTTP/2
  trailers, and if Hurl's libcurl binding can't surface them cleanly, the
  no-crates story gets harder. I put it as the first thing to validate.
  2. Descriptor sets vs .proto parser — accepting protoc -o foo.pb output as v1
  input could let us skip writing a .proto text parser entirely, since decoding
  a FileDescriptorSet is just protobuf decoding (which we have to write anyway).
   The plan suggests this as the v1 path with the text parser deferred.

⏺ Honest take — there's something I like, and one thing that bugs me.

  What's nice: putting the body intent next to the body itself (rather than only
   in [Options]) is a good instinct. It's also a natural extension point:
  ```prototext could later mean "this is proto text format, not JSON," and a
  future binary case could use file,. The fence becomes the discriminator for
  how to read this body.

  What bugs me: the fence is labeled protobuf but the content inside is JSON.
  That's a mismatch between the label (a wire format) and the content (a
  syntax). Hurl's existing fenced bodies follow the convention that the label
  names the content language — ```graphql really is GraphQL syntax. By that
  rule, the honest label here is ```json (or protojson if we want to be specific
   about which JSON encoding — Google's canonical proto-JSON mapping is
  officially called "protojson"). The "it'll be sent as protobuf wire bytes"
  part is the transport decision, which belongs in [Options] (grpc: true) — not
  on the body fence.

  There's also a redundancy question: with both grpc: true and ```protobuf,
  which one is load-bearing? If both, they have to stay consistent (one more
  thing to validate). If the fence drives it, grpc: true is dead weight; if
  grpc: true drives it, the fence is decorative. Pick one signal.

  content, and keep grpc: true as the single "this is gRPC" switch:

  POST localhost:50051/helloworld.Greeter/SayHello

  grpc: true
  proto: proto/helloworld.proto
  ```json
  {
    "name": "World"
  }
  ```

  HTTP 200

  grpc-status == 0

❯ With `buf curl` we have to set `--protocol grpc` to make it works. What's the 
   difference between connect and grpc?                                         

⏺ buf curl is a multi-protocol client — its default is Connect (a protocol Buf
  themselves invented), not real gRPC. --protocol grpc switches it to the actual
   gRPC-over-HTTP/2 wire format.

  The two protocols solve the same problem (typed RPC over HTTP) but with very
  different design tradeoffs:

  Column 1: Transport
  gRPC: HTTP/2 required
  Connect: HTTP/1.1 or HTTP/2
  ────────────────────────────────────────
  Column 1: Body framing
  gRPC: Length-prefixed protobuf frames (5-byte header + bytes)
  Connect: Plain protobuf or JSON body for unary; framed only for streams
  ────────────────────────────────────────
  Column 1: Status delivery
  gRPC: Mandatory HTTP/2 trailers (grpc-status, grpc-message)
  Connect: HTTP response code + JSON error body for unary; framed status for
    streams
  ────────────────────────────────────────
  Column 1: Content-type
  gRPC: application/grpc[+proto/+json]
  Connect: application/proto or application/json
  ────────────────────────────────────────
  Column 1: Curl-friendly?
  gRPC: No (needs HTTP/2 + manual gRPC framing — like we just did)
  Connect: Yes — plain curl -X POST -H 'content-type: application/json' -d 
    '{...}' works

  The Connect protocol exists because people found gRPC's reliance on HTTP/2
  trailers awkward — CDNs, browser clients (fetch/XHR), and HTTP/1.1 load
  balancers don't deal with trailers well. Connect's unary path is essentially
  "POST a JSON body, get a JSON body back, status code in the HTTP status." A
  third sibling, gRPC-Web, encodes the trailers into the body so HTTP/1.1
  clients can read them — that's what browsers typically use.

  buf curl speaks all three on the client side. Servers built with Connect-Go
  often expose all three on the same port. Your Python grpcio server only speaks
   real gRPC, which is why --protocol grpc is required.

  Relevance to the Hurl POC: we're targeting real gRPC (HTTP/2 + trailers — the
  spec at the link in §3). Connect and gRPC-Web are out of scope. Worth a
  one-liner in the PLAN's buf curl row to remind future-us that we should always
   pass --protocol grpc when running the survey. Want me to add it?

  design that today, but the fence shape leaves room.

  protobuf as the fence, that case and a gRPC call are indistinguishable. With

❯ I want to decode a protoset file, what's the reference documentation spec I   
  need to llok at?                                                              
⏺ You need two specs working together — one tells you how bytes encode, the
  other tells you what fields mean:

  1. The protobuf wire format — how bytes encode

  https://protobuf.dev/programming-guides/encoding

  This is the foundation. It defines:
  - Varint encoding (used for tags, lengths, ints, bools, enums)
  - Wire types (0=varint, 1=64-bit, 2=length-delimited, 5=32-bit)
  - Field tags = (field_number << 3) | wire_type
  - ZigZag for sint32/sint64
  - Length-delimited framing for strings, bytes, embedded messages, packed
  repeated

  Without this, you can't extract any field from any protobuf message —
  including a FileDescriptorSet.

  2. The descriptor schema — what the fields mean

  https://github.com/protocolbuffers/protobuf/blob/main/src/google/protobuf/desc
  riptor.proto

  This is Google's .proto file that defines the FileDescriptorSet message. It
  tells you field numbers, types, and nesting:

  FileDescriptorSet
  └── repeated FileDescriptorProto file = 1;

  FileDescriptorProto
  ├── string name = 1;          // e.g. "helloworld.proto"
  ├── string package = 2;       // e.g. "helloworld"
  ├── repeated DescriptorProto       message_type = 4;
  ├── repeated EnumDescriptorProto   enum_type    = 5;
  ├── repeated ServiceDescriptorProto service    = 6;
  └── ...

  DescriptorProto
  ├── string name = 1;
  ├── repeated FieldDescriptorProto field = 2;
  └── ...

  ServiceDescriptorProto
  ├── string name = 1;
  └── repeated MethodDescriptorProto method = 2;

  MethodDescriptorProto
  ├── string name = 1;
  ├── string input_type  = 2;
  ├── string output_type = 3;
  ├── bool   client_streaming = 5;
  └── bool   server_streaming = 6;

  You don't need to decode all of descriptor.proto — only the field numbers Hurl
   actually consumes (service, method, message, field, type, label, enum).
  Everything else can be safely skipped as unknown fields per the protobuf
  forward-compatibility rules.

  Optional context

  - proto3 language guide (helpful for understanding defaults, oneof, map,
  etc.): https://protobuf.dev/programming-guides/proto3
  - gRPC-over-HTTP/2 protocol (for the framing layer that wraps these protobuf
  bytes when we actually make a call):
  https://github.com/grpc/grpc/blob/master/doc/PROTOCOL-HTTP2.md

  Practical reading order

  1. Read encoding spec (1) end-to-end — it's short (~10 min) and
  self-contained.
  2. Skim descriptor.proto (2) — find the fields you actually need.
  3. Start implementing: varint decoder → tag parser → length-delimited reader →
   FileDescriptorSet walker.

  Both links are already in client/README.md's References section, so you're set
   to bookmark from there.

⏺ Quick architecture take first: splitting reader.rs (wire layer) from
  descriptor.rs (schema layer) is the right shape, the Self::parse(&[u8]) -> 
  Result<Self, _> pattern is idiomatic, and _ => reader.skip(wire_type)?
  correctly implements protobuf's forward-compat rule. Solid foundation. Below
  are the issues I'd address — grouped by what matters.

  ---
  Bugs / correctness (fix these)

  1. Reader::read_bytes can panic on a hostile input

  fn read_bytes(&mut self, n: usize) -> &[u8] {
      let start = self.pos.0;
      self.pos.0 += n;
      &self.input[start..start + n]   // ← panics if n overshoots input
  }

  n comes from read_varint() in read_len_delimited, which reads untrusted bytes.
   A truncated or malicious .protoset (or even one we accidentally pass with a
  bug elsewhere) will panic instead of returning an error. Should return
  Result<&[u8], ReaderError> and check start + n <= input.len(). Same issue in
  skip(WireType::Len) and skip(WireType::I64/I32) — none of them bounds-check
  before advancing pos.

  2. read_varint has no maximum-length guard

  loop {
      let byte = match self.read_byte() { ... };
      result |= ((byte & 0x7F) as u64) << shift;
      if byte & 0x80 == 0 { break; }
      shift += 7;
  }

  A valid protobuf varint is at most 10 bytes (64 bits / 7 bits per byte,
  rounded up). After 10 continuation bytes, shift >= 70 and the << becomes
  UB-ish (Rust panics in debug, wraps in release). Add a guard: after 10 bytes,
  return ReaderError::InvalidVarint (new variant).

  3. panic!("Inconsistent type and type_name") in library code

  if field_type != FieldType::Group && ... {
      panic!("Inconsistent type and type_name")
  }

  Same class as #1 — input-driven panic in a library. Should be a ReaderError
  variant. Also: the cross-field consistency check probably belongs at a higher
  level than the decoder (you could validly want to decode a
  malformed-but-readable descriptor and report on it). I'd consider dropping the
   check entirely from parse — leave it to a separate "validate" pass.

  4. read_int32 returns u32 but protobuf int32 is signed

  pub fn read_int32(&mut self) -> Result<u32, ReaderError> {
      let value = self.read_varint()?;
      u32::try_from(value).map_err(|_| ReaderError::InvalidInt32)
  }

  In protobuf terminology, int32 is a signed 32-bit varint (negative values get
  sign-extended to a 10-byte varint). What you've written is uint32. Today this
  works because field numbers are always positive, but it's a footgun the moment
   someone reuses the function on another int32 field that can be negative.

  Two options:
  - Rename to read_uint32 and add a proper read_int32 -> i32 later.
  - Or make it pub fn read_int32(&mut self) -> Result<i32, _> now and use as i32
   (bit-cast from the varint's low 32 bits).

  5. read_string allocates twice

  let bytes = self.read_len_delimited()?;
  String::from_utf8(bytes.to_vec()).map_err(|_| ReaderError::InvalidUtf8Bytes)

  bytes.to_vec() allocates, then from_utf8 validates. Replace with:

  std::str::from_utf8(self.read_len_delimited()?)
      .map(str::to_owned)
      .map_err(|_| ReaderError::InvalidUtf8Bytes)

  One allocation, same semantics.

  ---
  API / idiom

  6. ReaderError doesn't implement std::error::Error

  You have Debug + Display but not Error. Hurl's eventual integration will want
  to compose this with ? into a larger error type — implement Error (the body
  can stay empty since you have no source()). One-liner:

  impl std::error::Error for ReaderError {}

  7. ReaderError::Generic is a smell

  Used as the fallback in FieldType::try_from for an unknown enum value. Replace
   with a typed variant like InvalidFieldType { value: u64 } that carries the
  offending value — much easier to debug.

  8. Display for ReaderError is half-finished

  ReaderError::Eof => write!(f, "ReaderError::Eof"),
  ReaderError::InvalidWireType { .. } => write!(f,
  "ReaderError::InvalidWireType"),

  These leak the type name and drop the captured context. The InvalidField
  variant shows the right shape — every other one should follow it:

  ReaderError::Eof => write!(f, "unexpected end of input"),
  ReaderError::InvalidWireType { wire_type } => write!(f, "invalid wire type: 
  {wire_type}"),
  ReaderError::LegacyWireType { wire_type } => write!(f, "unsupported 
  (deprecated) wire type: {wire_type}"),
  ReaderError::InvalidBool => write!(f, "invalid bool (varint was neither 0 nor 
  1)"),

  9. BytePos(usize) newtype is dead weight

  It's only ever accessed via .0 and only ever incremented. Either give it an
  API (Add<usize>, advance, Display for error context) or drop it for a plain
  usize. As written, it pays the cost of a newtype with none of the benefits.

  10. Open-coded self.pos.0 += n everywhere is the natural home for #1

  A single fn advance(&mut self, n: usize) -> Result<(), ReaderError> that
  bounds-checks would fix the panics in #1 in one place, and let you delete the
  duplicated self.pos.0 += ... lines in read_bytes, skip, etc.

  11. Field-number magic constant deserves a name

  if field_number == 0 || field_number > 0x1FFF_FFFF {

  (1 << 29) - 1 is the max protobuf field number — give it a name and a link to

  /// Max protobuf field number — `2^29 - 1`. See encoding spec.

  FileDescriptorSet, FileDescriptorProto, DescriptorProto, etc. all derive

  Drop the derives.

  ---
  On domain separation

  The leak in your current design isn't structural — it's in the error type.
  ReaderError is doing two jobs:

  Wire-level (Reader's job): Eof, BufferOverflow
  Schema-level (Descriptor's job): InvalidField { entity, field, ... }
  ────────────────────────────────────────
  Wire-level (Reader's job): InvalidVarInt, InvalidWireType
  Schema-level (Descriptor's job): UnsupportedSyntax { syntax }
  ────────────────────────────────────────
  Wire-level (Reader's job): InvalidFieldNumber, LegacyWireType
  Schema-level (Descriptor's job): Generic (used for "unknown FieldType enum
    value")
  ────────────────────────────────────────
  Wire-level (Reader's job): InvalidUtf8Bytes, InvalidBool, InvalidInt32
  Schema-level (Descriptor's job): the (currently panicking) "Inconsistent type
    and type_name" check

  The reader has no business knowing what an "entity" or a "field name" is —
  those are schema concepts. Mixing them is why the boundary feels fuzzy.

  Concrete split:

  // reader.rs — purely wire-level, no schema concepts
  pub enum ReaderError {
      Eof,
      BufferOverflow,
      InvalidVarInt,
      InvalidWireType(u8),
      LegacyWireType(WireType),
      InvalidFieldNumber(u64),
      InvalidUtf8(std::str::Utf8Error),
      InvalidBool(u64),
      IntegerOverflow,           // formerly InvalidInt32
  }

  // descriptor.rs — schema-level, wraps wire errors
  pub enum ParseError {
      Reader(ReaderError),       // wire-level passthrough
      UnexpectedWireType { entity: &'static str, field: &'static str,
                           expected: WireType, actual: WireType },
      UnknownFieldType(u64),
      UnsupportedSyntax(String),
      InconsistentTypeAndTypeName,
  }

  impl From<ReaderError> for ParseError {
      fn from(e: ReaderError) -> Self { ParseError::Reader(e) }
  }

  This buys you:
  - Sharp boundary. reader.rs is reusable for any protobuf message, not just
  descriptors. The day Hurl needs to decode a HelloReply on the wire (not a
  descriptor), it uses the same Reader with no schema-tinted error baggage.
  - Better diagnostics. A user reading an error stops asking "is this a corrupt
  protoset or a schema my code doesn't handle?" — the error type tells them.
  - ? still works because of From<ReaderError> for ParseError.

  Honest counter-argument: the two-error split has a cost in From glue and
  slightly noisier error matching. For a tiny POC you could leave them merged.
  But you specifically said the domains feel muddy — splitting the errors is the
   lever that fixes that perception in code, not just in your head.

  ---

⏺ The _typed suffix is slightly misleading — the base read_string() is already
  "typed" (returns String). What these new methods actually add is a wire-type 
  check before the read. So a better name reflects that, not "typing."

  Ranked from most to least idiomatic Rust:

  1. expect_string(wire_type) — prefix swap (my pick)

  let name = reader.expect_string(wire_type)?;
  let n    = reader.expect_uint32(wire_type)?;
  let flag = reader.expect_bool(wire_type)?;
  let bytes = reader.expect_message(wire_type)?;

  - Reads as English: "I expect this to be a string given the tag I just read."
  - Mirrors Option::expect / Result::expect in spirit (an assertion that returns
   an error instead of panicking).
  - Short, prefix is the natural place for "kind of operation" in Rust.

  2. read_string_checked(wire_type) — suffix that matches std

  let name = reader.read_string_checked(wire_type)?;

  - Matches the std convention for "validating variants" (checked_add,
  checked_sub, try_into family).
  - Says exactly what it does: same read, with a check.
  - A bit longer but unambiguous.

  3. Just string(wire_type, "field") — only if you go with FieldReader

  let name = fr.string(wire_type, "name")?;
  let n    = fr.uint32(wire_type, "number")?;
  let bytes = fr.message(wire_type, "field", FieldDescriptorProto::parse)?;

  - On a FieldReader newtype the read_ prefix becomes redundant (the whole type
  is a reader).
  - Shortest possible; arms read like a translation of the .proto.
  - Only works if you commit to the FieldReader wrapper — on bare Reader,
  reader.string(...) is too vague.

  Suffixes I'd avoid

  - _typed — current; suggests typing where the real addition is checking.
  - _strict — implies the loose variant is the default, which is backwards (the
  loose one wouldn't exist).
  - _validated — too long and slightly bureaucratic.
  - _at_wire / _with_wire — awkward English.

  My concrete suggestion

  Use expect_* on bare Reader (it's the cleanest "I assert + read" name in
  Rust). If you go all the way to the FieldReader wrapper, drop the prefix
  entirely and use bare verbs — the type itself supplies the context. Don't
  bother with a suffix at all.

❯ In the Rust client, we have build a FileDescriporSet that is an AST for       
  protobuf descriptor. I would like to construct a kind of Symbol table which   
  will keep fully qualified name and enum, message, service, method descriptor. 
   What would you propose to achieve this (includeing structs change if needed) 

⏺ Here's a concrete proposal, with the parts split between "AST plumbing you
  need first" and "the symbol table itself."

  1. AST changes you need first

  Before a symbol table is meaningful, the descriptor tree has to actually
  represent everything you want to look up. Today it's partial:

  - FileDescriptorProto only has message_types. Add enum_types: 
  Vec<EnumDescriptorProto> (top-level enums) and services: 
  Vec<ServiceDescriptorProto>.
  - DescriptorProto has nested_types but not enum_types. Add enum_types: 
  Vec<EnumDescriptorProto> for nested enums (your Payload.Priority).
  - EnumDescriptorProto is currently {}. Flesh it out — at minimum name and
  values: Vec<EnumValueDescriptorProto>.
  - Add ServiceDescriptorProto { name, methods: Vec<MethodDescriptorProto> } and
   MethodDescriptorProto { name, input_type: String, output_type: String, 
  client_streaming: bool, server_streaming: bool }. The two _type fields are
  exactly the FQNs the symbol table resolves.
  - Make struct fields pub (these are inert data carriers). Or keep them private
   and put the symbol table inside descriptor.rs — either is defensible. I'd go
  pub and put the symbol table in a separate module, so the wire-decoder and the
   indexing concerns don't share a file.

  2. Symbol table design — borrowed, single map

  // src/symbols.rs
  use std::collections::HashMap;
  use crate::descriptor::{
      DescriptorProto, EnumDescriptorProto, FileDescriptorSet,
      MethodDescriptorProto, ServiceDescriptorProto,
  };

  pub struct SymbolTable<'a> {
      by_fqn: HashMap<String, Symbol<'a>>,
  }

  #[derive(Copy, Clone)]
  pub enum Symbol<'a> {
      Message(&'a DescriptorProto),
      Enum(&'a EnumDescriptorProto),
      Service(&'a ServiceDescriptorProto),
      Method {
          service: &'a ServiceDescriptorProto,
          method:  &'a MethodDescriptorProto,
      },
  }

  #[derive(Debug)]
  pub enum SymbolError {
      Duplicate { fqn: String },
  }

  Key design choices, with reasoning:

  - Borrowed (&'a everywhere), no clones. The symbol table is a read-only index
  into a FileDescriptorSet you've already parsed. There's no reason to copy. The
   lifetime constraint isn't a problem in practice — you build the table right
  after parsing and use it for the duration of the request.
  - Single HashMap<String, Symbol<'a>> keyed by FQN. Protobuf rules already
  guarantee sibling-type names are unique within a scope, so FQNs are globally
  unique across the whole set. One map = one lookup = simplest possible. Use
  BTreeMap instead only if you want deterministic dumps for tests.
  - FQNs stored without the leading . Descriptor type-name strings carry a
  leading dot (.echo.Payload) but it's just a "this is absolute" marker.
  Normalize at insert and at lookup so the caller can pass either form:
  fn norm(fqn: &str) -> &str { fqn.strip_prefix('.').unwrap_or(fqn) }
  - Method is a variant, not a top-level kind. A method's FQN (echo.Echo.Echo)
  shares the namespace with everything else, but you almost always want to know
  which service it belongs to — packaging the parent service in the variant
  saves a second lookup.
  - Field lookups stay on the message, not in the table. Fields are scoped to
  their parent message and you typically iterate msg.fields to find one. Adding
  them globally just bloats the table.

  3. Build algorithm

  Recursive walk; each frame carries a "parent FQN" string that you concatenate
  as you descend:

  impl<'a> SymbolTable<'a> {
      pub fn build(fds: &'a FileDescriptorSet) -> Result<Self, SymbolError> {
          let mut by_fqn = HashMap::new();
          for file in &fds.files {
              let pkg = file.package.as_deref().unwrap_or("");
              for msg  in &file.message_types { add_message(&mut by_fqn, pkg,
  msg)?; }
              for en   in &file.enum_types    { add_enum   (&mut by_fqn, pkg, en
   )?; }
              for svc  in &file.services      { add_service(&mut by_fqn, pkg,
  svc)?; }
          }
          Ok(SymbolTable { by_fqn })
      }
  }

  fn join(parent: &str, name: &str) -> String {
      if parent.is_empty() { name.to_string() } else {
  format!("{parent}.{name}") }
  }

  fn add_message<'a>(
      map: &mut HashMap<String, Symbol<'a>>, parent: &str, msg: &'a
  DescriptorProto,
  ) -> Result<(), SymbolError> {
      let fqn = join(parent, msg.name.as_deref().unwrap_or(""));
      insert(map, fqn.clone(), Symbol::Message(msg))?;
      for nested in &msg.nested_types { add_message(map, &fqn, nested)?; }
      for en     in &msg.enum_types   { add_enum   (map, &fqn, en    )?; }
      Ok(())
  }

  fn add_service<'a>(
      map: &mut HashMap<String, Symbol<'a>>, parent: &str, svc: &'a
  ServiceDescriptorProto,
  ) -> Result<(), SymbolError> {
      let svc_fqn = join(parent, &svc.name);
      insert(map, svc_fqn.clone(), Symbol::Service(svc))?;
      for m in &svc.methods {
          let m_fqn = join(&svc_fqn, &m.name);
          insert(map, m_fqn, Symbol::Method { service: svc, method: m })?;
      }
      Ok(())
  }

  fn insert<'a>(
      map: &mut HashMap<String, Symbol<'a>>, fqn: String, sym: Symbol<'a>,
  ) -> Result<(), SymbolError> {
      if map.insert(fqn.clone(), sym).is_some() {
          return Err(SymbolError::Duplicate { fqn });
      }
      Ok(())
  }

  4. Public API

  Keep it small:

  impl<'a> SymbolTable<'a> {
      pub fn lookup(&self, fqn: &str) -> Option<Symbol<'a>>;

      // Typed sugar — returns None if FQN isn't that kind:
      pub fn message(&self, fqn: &str) -> Option<&'a DescriptorProto>;
      pub fn enum_type(&self, fqn: &str) -> Option<&'a EnumDescriptorProto>;
      pub fn service(&self, fqn: &str) -> Option<&'a ServiceDescriptorProto>;
      pub fn method(&self, fqn: &str) -> Option<(&'a ServiceDescriptorProto, &'a
   MethodDescriptorProto)>;

      // Iteration (handy for "list all services" UX):
      pub fn services(&self) -> impl Iterator<Item = (&str, &'a
  ServiceDescriptorProto)> + '_;
  }

  Symbol is Copy because it's pointers, so callers can stash one cheaply.

  5. Alternatives I considered and rejected

  Approach: Owned (clone descriptors into the table)
  Why I'd skip it: Doubles memory and creates a second source of truth that can
    drift from the AST. The lifetime "problem" it sidesteps doesn't actually
    exist in your use case.
  ────────────────────────────────────────
  Approach: Indices instead of &'a (e.g. MessageId { file: usize, path: 
    Vec<usize> })
  Why I'd skip it: More flexible (the table can outlive any specific borrow) but

    you pay a real cost: every lookup is two-step, and the path: Vec<usize> is
    harder to read in debug output than a borrowed reference. Worth it only if
    you need to mutate the AST after building the table, which you don't.
  ────────────────────────────────────────
  Approach: Arena allocator (id-arena etc.)
  Why I'd skip it: Solves a problem you don't have (cycles, frequent
    insert/delete) at the cost of dragging in tooling.
  ────────────────────────────────────────
  Approach: One map per kind (messages, enums, services, methods)
  Why I'd skip it: More typed but four times the lookup code, and the
  lookup(fqn)
    -> Symbol API becomes "try each map in turn." Single map + enum tag is
    shorter and equivalent.
